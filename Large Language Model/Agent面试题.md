# 什么是 Agent ？与大模型有什么本质不同？



>  我理解的 Agent 本质上是一个能自主完成目标的 AI 系统，跟传统 AI 最核心的区别在于「自主性」和「能行动」。传统 AI 是你问一个问题它回答一个问题，每次都是独立的，被动响应的；而 Agent 有自己的规划能力，你给他一个复杂的目标，它会把任务拆成多步，通过调用工具、访问记忆、感知环境来一步步执行，直到完成。它不只是输出文字，而是真的能做事。



说到与大模型的区别，那就得先说说大模型的局限性了。

大模型的知识是死的，它的知识是训练成的，也就是会有一个截止日期，在这个日期之后的知识大模型都是不知道的；此外，大模型的手脚是没有的，大模型本质上还是一个一问一答的机器，你叫它去干点事情，它只会说"你可以这样做"，但是它始终都不会动；然后就是大模型的记忆是断的，每一次的提问，大模型都不会记得上次的聊天内容，除非手动添加聊天历史。

这就意味着普通 LLM 只能做「一问一答」的事情，稍微复杂一点的、需要多步骤协作的任务，它就完全无能为力了



Agent 就完全不一样了。它有一个核心的运作闭环：**感知 -> 规划 -> 行动 -> 再感知**。

**Agent能自主完成目标的核心技术就是 工具调用、记忆机制、多步推理和自我纠错。**



面试时答这道题，一定要点出三件事：一是 Agent 有自主规划能力，给它一个复杂目标它能自己拆解成多步；二是它能行动，通过工具调用跟外部世界真实交互；三是它有闭环，每步的结果会反馈回来指导下一步，而不是一次性生成完就结束。另外还要提一句容易混的点：模型本身只是「大脑」，工具的真正执行是你的代码，模型只负责决策

# Agent 的基本架构由哪些核心组件构成？



Agent 的基本架构由哪些核心组件构成？
怎么回答这个问题呢？
一个Agent的正常运行有哪些功能？
LLM 当 Agent 的大脑，工具调用系统当 Agent 的手脚，还有什么呢？
Agent 的一个最小运行单元是啥？是think-execute，感知、规划、行动，感知依赖什么？记忆机制啊！
所以现在就有三个核心组件了：LLM、工具调用系统、记忆机制；
还有一个核心组件就是规划模块。



>根据看资料我了解到的 Agent 基本架构的核心组件有四个：LLM、记忆机制、工具调用系统、规划模块；LLM是整个系统的大脑，负责理解任务和做出决策；工具让 Agent 能跟外部世界交互，搜索、执行代码、调 API 都靠它；记忆让 Agent 在任务执行过程中保持状态，不会「失忆」；规划模块负责把复杂目标拆解成可执行的步骤。这四个组合在一起，才让 Agent 具备了自主完成任务的能力。
>
>这四个核心组件都可以追溯到 Agent 的运行闭环上。感知阶段中，Agent 需要LLM进行思考，还需要记忆来提供之前的聊天内容；规划阶段，Agent 需要进行规划，也就是规划模块；执行阶段，Agent 需要工具调用系统，才可以真正动起来。



规划模块其实依赖的是大模型的推理能力，规划模块在实际运作中有两种主流模式，第一种是先规划后执行，也就是`Plan-and-Execute`模式；另外一种是边执行边规划，也就是 ReAct 模式。





# Workflow、Agent、Tools这三个概念和区别介绍一下？

>我理解的这三个概念是颗粒度从小到大的三层结构。Tools是最小的能力单元，就是封装好的可调用函数，比如搜索、执行代码、发邮件、它只负责执行，本身没有任何决策能力；Agent 是一个完整的决策系统，内部使用LLM作为大脑，自己判断什么时候调用哪个Tool、要不要继续、什么时候结束，是主动的；Workflow是更上层的编排框架，把Agent、LLM、Tools组织成一条确定性流程，每个节点做什么、按照什么顺序流转都是开发者事先写死的。三者最核心的区别就是：Tools不做决策只执行，Agent 自己做决策，Workflow 是开发者替所有节点把决策提前写好。



这三个不是同一维度的东西，而是粒度不同、可以相互嵌套的三层结构。在实际做项目的时候，三者通常都是同时存在，只是扮演不同的角色。



`Tools`是一个封装好的函数，有明确的输入参数和明确的输出结果。但是`Tools`和普通函数并不同，`Tools`除了函数本身外，还需要提供一份“说明书”。这份“说明书”是给LLM看的。说明书中需要声明工具叫什么、能做什么事、需要传入哪些参数，这样LLM才能知道自己有哪些能力可以调用。

注意一个设计关键：**工具本身没有任何的决策能力，这个工具甚至都不知道自己应该什么时候被调用。**这并不是设计缺陷，而是故意的，Tools 的使命就是把一个具体的能力封装好，随时待命，至于什么时候被使用，这不是自己的事。



工具有几个设计原则：职责单一、描述要精确、错误信息要清晰、参数设计要简洁。





# 了解哪些其他的 Agent 设计范式？Agent 和 Workflow 的区别是什么？



> 我理解的 Agent 和 Workflow 的最核心区别是谁来决定下一步， Workflow 是开发者提前把流程写好了，每一步怎么走都是固定的，确定性高，好控制、调试；Agent 是让 LLM 自己决定下一步做什么，灵活但是不可控。
>
> 常见的设计范式除了纯 Agent 之外，还有 ReAct、Plan-and-Execute、Reflection 这几种。我在实际工程中用得最多的反而是把两者混用，固定流程的部分使用 Workflow，需要灵活决策的节点嵌入 Agent 能力。



## 三种设计范式

### ReAct 

ReAct 是边思考边执行，也就是把思考和执行交替进行。ReAct 的每一轮循环都由三个步骤组成，形成一个完整的 Thought -> Action -> Observation 循环。



不过 ReAct 也有一个明显的短板：它是「走一步看一步」的模式，每一步都是局部最优决策，处理特别复杂的、需要全局规划的任务时，容易在中间迷失方向。比如一个需要十几步才能完成的研究任务，ReAct 可能做到第五步就忘了最初的目标是什么，或者反复在几个工具之间打转。



### Plan-and -Execute

Plan-and-Execute 把规划和执行彻底分开，先让一个 LLM 专门做规划，输出一个完整的步骤列表，然后由另外一个 LLM 逐步执行。

**这里有一个非常关键的机制需要特别说一下：动态重规划。很多人对 Plan-and-Execute 的理解停留在「先做一个计划，然后死板地执行」，这其实是不对的。**

成熟的 Plan-and-Execute 实现里，每执行完一步都会把结果反馈给规划器，规划器会判断：当前的执行结果和预期一致吗？后续的计划还适用吗？需不需要调整？如果发现某一步的结果和预期严重偏离，规划器会修改后续的步骤，甚至插入新的步骤来应对。



### Reflection

Reflection 则是在前两种范式的基础上加了一层「质量保障」。它的做法是在 Agent 完成一步或者完成整个任务之后，再让一个 LLM（可以是同一个模型也可以是专门的评估模型）来判断做得好不好、结果是否符合预期。如果评估不通过，就重试或者换一种策略。这个机制能显著提升输出质量，尤其是在代码生成、文案写作这类「质量要求高但一次做对很难」的场景下效果特别明显。



### 三种范式怎么选择

那面试的时候如果被追问「这三种范式怎么选」，怎么回答？其实核心看两个维度：任务复杂度和质量要求。如果任务步骤不多、每步都比较独立，ReAct 就够了，简单直接；如果任务很复杂、步骤之间有依赖关系需要全局统筹，Plan-and-Execute 更合适；如果对输出质量要求特别高、允许多花一些时间和成本，就在前面的基础上叠加 Reflection。实际项目里这三种范式也不是互斥的，很多系统会混合使用，比如用 Plan-and-Execute 做整体规划，每个步骤内部用 ReAct 来执行，关键步骤再加上 Reflection 做质量把关。





# Agent 推理模式有哪些？ReAct是啥？具体是怎么实现的？

> Agent 的推理模式我用过下面几种：
>
> 最基础的就是直接交付答案，没有中间的推理过程；CoT 是让 LLM 先把推理过程写出来再给答案，准确率更高，；ReAct 是在 CoT 的基础上加了 行动，让  LLM 交替输出思考和工具调用，每次行动后再根据结果继续思考，**形成一个循环**；
>
> 我认为 ReAct 是目前 Agent 用得最广的模式，它的推理过程可见，又能动态利用外部工具，两个优点都有



在继续这个问题前，我查了一下，ReAct 既可以是 Agent 的一种推理模式，也可以是一种设计范式。当是推理模式时，强调的是记录，记录的是思考和行动的过程，当是设计范式时，侧重的是循环流程，Thought -> Action -> Observation



推理模式存在的根本原因就是：通过不同的方式，让 LLM 把隐式的思考过程显式化出来，从而减少多步推理中累积的误差。CoT、ReAct 就是这个方向上的两种解法，每一个都在解决前一个的局限性。



### CoT的两种触发方式

1. Zero-shot CoT

   这种方式的做法非常简单，在 Prompt 的末尾加上 “让我们一步步思考” 这句话就行了，LLM 会自己展开推理过程，不需要我们提供任何示例。这种方式的优势是零成本、即插即用，但是缺点是 LLM 的推理格式和深度完全是依靠 LLM 自己的发挥，不太稳定，有时候会写得很详细，有时候又跳步。

   

2. Few-shot CoT

   你需要在 prompt 里给几个带有完整推理过程的例子，让 LLM 照着这个格式来模仿。Few-shot 的效果更稳定，特别适合输出格式要求比较固定的场景，代价是需要你提前准备高质量的示例，而且示例本身也会占用 token。



但 **CoT 有一个根本性的局限：它是纯文字推理，没有办法和外部世界交互。**推理过程再完整，也拿不到实时数据，不能执行计算，不能访问数据库。





# ReAct、Plan-and-Execute、Reflection 三种范式有什么核心区别？实际项目中该如何选型？

> 这三者是 Agent 开发中最主流的三种设计范式，核心区别就在于 **“决策和执行的关系”**
>
> ReAct 是边想边干，单步迭代实时调整，灵活度最高，适合于简单任务；
> Plan-and-Execute 是先想全再干，先定制完整的计划再分步执行，适合长流程复杂任务，不容易跑偏；
> Reflection 不是独立的完整的流程，而是给前两者加的 “检查修正buff”，用来提升输出质量。
>
> 实际选型主要看三个维度：任务复杂度、流程确定性、输出质量要求。



## 设计范式和推理模式的区别

设计范式：就是 Agent 系统顶层的做事流程。

推理模式：Agent 每一步干活时 脑子里具体是怎么思考的。



## ReAct - 单步迭代范式

ReAct 是所有 Agent 范式的祖宗，它的本质就是 思考 -> 行动 -> 观察 -> 再思考 的一个循环，走一步看一步，每一步的行动都完全基于于上一步的结果进行实时调整，没有提前定死的完整计划。

ReAct 和另外两种范式最核心的区别就是 **它没有提前做完整规划 的环节，规划和执行是混在一起的。**而 **Plan-and-Execute 是把两者完全拆开，先做完整的规划再统一执行**。**和 Reflection 比较，ReAct 的循环里没有专门的自我检查环节**，只有行动后的结果观察，不会停下来复盘这一步做对没有。



ReAct 最大的优势是实现简单、灵活度高、逻辑透明，出了问题好排查，新手入门零门槛。但它的短板也很明显：遇到长流程、多步骤的复杂任务，很容易走着走着就跑偏，忘了最初的目标，也容易在某一步陷入无效循环。所以它更适合流程不固定、复杂度适中的任务，比如日常信息搜索、简单问答助手、客服机器人，也是新手入门的首选。



## Plan-and-Execute 规划执行范式

Plan-and-Execute 是针对 ReAct 长任务容易跑偏 的痛点做的针对性优化，ReAct 是走一步看一步，边想边干；Plan-and-Execute 是先把完整计划制定好，再按计划一步步干。这是两种不同的思路。



适用于需要任务流程清晰、严格控制方向的场景。





## Reflection 反思迭代范式

**Reflection 不是一套独立的完整流程，而是给 ReAct、Plan-and-Execute 加的「锦上添花的 buff」**，**它不改变原本的做事流程，只是在原本的基础上，加了一层「自我检查、自我修正」的环节。**

Reflection 和前两者最本质的区别，是它不是一套独立的做事流程，而是可以叠加在 ReAct 或 Plan-and-Execute 之上的增强机制，互不冲突。前两者的核心是「把事做完」，Reflection 的核心是「把事做好」，专门解决输出质量不达标、有事实错误、逻辑漏洞的问题。





实际项目中还有一种非常常见的混合用法：规划阶段用 Plan-and-Execute 的思路定好全局计划，每一步的执行用 ReAct 的循环来处理（因为单步执行可能也需要多轮工具调用），最后对整体输出做一次 Reflection 检查质量。这种三层嵌套的架构听起来复杂，但其实在 LangGraph 这类框架里实现起来很自然，很多生产级的 Agent 系统都是这么搭的。





# 复杂任务怎么做的任务拆分？为什么要拆分？效果提升如何？

> 复杂任务拆分有两种形式：一种是静态拆分，提前把步骤写死；另外一种是动态拆分，让LMM自己根据目标规划步骤，更灵活但也更难控制；
> 把复杂任务做拆分的原因是：LLM 一次性处理太复杂的任务很容易出错，把大任务拆分成小任务，每一步聚焦一件事，准确率会更高，但本质上还是因为LLM 的上下文窗口有大小限制，一次步骤需要做的事情太多的话，上下文窗口可能不够大，会导致做着做着不记得前面的事；
>
> 如果拆完之后步骤之间有依赖关系，可以把能并行的步骤并发跑，端到端延迟可以降很多，40%~60%。



拆分的本质原因是 LLM 的工作台，也就是 context window，是有大小限制的，**能同时处理的信息量是有上限的。**



但是任务不是拆得越细越好，任务拆得细会暴露出两个代价：步骤变多，LLM 的调用次数就会变多，token 总消耗上升；步骤变多，每步只会做一件极小的事，LLM会看不到全局（因为每步执行时会把前面步骤的结果上传到上下文中），产出的各个部分也容易衔接生硬。

但是拆的步骤太少又会回到那个问题：每步负责的事太多，出错的概率上升，出了问题也不好定位。



# 介绍下AI Agent 的记忆机制，并说明在实际开发中应该如何设计记忆模块？

> 记忆机制分四层：感知记忆（当前输入的原始内容）、短期记忆（context window 里的对话历史）、长期记忆（存在外部数据库、语义检索召回）、实体记忆（结构化提取的关键事实）。
>
> 实际设计时要解决三个核心问题：存什么、怎么存、什么时候取出来用，根据信息类型选合适的存储方式，再搭配主动检索和按需检索两种策略使用。



**感知记忆**：这是最短暂的一层，就是「当前这次调用的原始输入」，用户发来的这条消息、上传的截图、传入的文档。它的生命周期只有一次调用，处理完就消失，不会主动保留。类比到人的话，就是你刚听到的一句话，如果没有主动去记，几秒后就忘了。感知记忆就是这个「刚进来还没处理」的原始感知，它存在的意义是给模型提供一个「入口」来接收外部信息。



**短期记忆**：这是 context window 里的 messages 列表，维持着当前任务执行过程中的完整状态，包括用户说了什么、模型输出了什么、工具调用返回了什么。只要任务还在进行，这些信息就都在；任务结束（对话关闭），这块记忆就清空了。



**长期记忆**：这是跨任务保留的信息，存在外部数据库里，通常是向量数据库、关系数据库或 Key-Value 存储。任务结束了，信息不会消失，下次需要时去检索拿回来用。

长期记忆可以细分成三类：

- 「情节记忆」（Episodic Memory），存的是具体的事件经历。比如「上周二用户让我写了一个 Python 爬虫，中间遇到了反爬问题，最后用 Selenium 解决了」，这是一段完整的任务经历，包含了时间、场景、过程和结果。情节记忆的价值在于，当 Agent 遇到类似的新任务时，可以检索出历史上的相似经历，参考上次是怎么解决的，避免重复踩坑。
- 「语义记忆」（Semantic Memory），存的是从多次经历中提炼出来的通用知识和规律。比如经历了好几次反爬问题之后，Agent 沉淀出一条规律：「当目标网站有 JavaScript 动态渲染时，requests 库抓不到内容，应该优先考虑 Selenium 或 Playwright」。这不再是某一次具体的事件记录，而是跨多次经验总结出来的抽象知识。语义记忆的信息密度更高，检索时也更容易命中，因为它直接存储的就是结论而不是过程。
- 「程序记忆」（Procedural Memory），存的是怎么做某件事的操作流程。比如「部署一个 Flask 应用的标准步骤：创建虚拟环境 -> 安装依赖 -> 配置 gunicorn -> 设置 nginx 反向代理 -> 启动服务」，这是一套可以直接复用的操作 SOP。程序记忆在处理重复性任务时特别有用，Agent 不需要每次都从头推理，直接调出对应的 SOP 执行就行，既快又稳。



**实体记忆**：这层比长期记忆更精炼，它不是存原文，而是把对话中出现的关键实体和事实主动提取出来，存成结构化字段。比如「用户偏好 Python」「客户预算是 5 万」「项目截止日是 3 月底」，这些是从对话里提炼出来的「结论」，而不是原始对话本身。类比到人的话，就像医生的病历卡，不是把问诊录音存起来，而是结构化地记录「主诉：头痛三天；诊断：偏头痛；用药：布洛芬」。信息密度高，查询快，而且不受原始表述方式影响。



## 你的项目中哪些地方使用了向量数据库来存储，哪些地方使用了关系型数据库存储

在我的项目中，agent 信息、消息体、消息会话、知识库对象（不包含知识库内容）、文档对象（不包括文档内容）使用的都是关系型数据库，但是文档内容，采用的是向量数据库，先把文档切片，然后使用bge-m3模型（把文字转化成向量）向量化存储到数据库中。



## Context Windows 管理

- 滑动窗口
- 摘要压缩
- 还有一种做法是把不常用但重要的信息「卸载」到长期记忆里。执行过程中产生的中间结果，如果当前步骤不需要但后面可能用到，就先存到向量数据库里，从 context window 中移除，等后面某步需要时再检索回来。这相当于给工作台配了一个「抽屉」，桌面放不下的东西先收到抽屉里，要用的时候再拿出来。



## 总结

回答 Agent 记忆机制这道题，先把四层分类说清楚：感知记忆是当次调用的原始输入，最短暂；短期记忆是 context window 里的 messages，维持任务状态；长期记忆是存在向量或关系数据库里、跨任务持久化的内容；实体记忆是从对话中提炼出来的结构化事实，信息密度最高。

说完分类，再答三个工程核心问题：存什么（只存对下次任务有价值的内容，过滤噪音）、怎么存（语义内容用向量数据库，结构化偏好用关系数据库，混合存储是主流）、什么时候取（任务开始前主动检索加载背景，执行中按需检索特定知识）。

最后用「读 -> 用 -> 写」三阶段闭环收尾，整个回答结构清晰、有深度，面试官很难挑剔。
