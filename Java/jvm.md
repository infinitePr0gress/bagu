## 什么是jvm？

jvm是java虚拟机，本质上是计算机中运行的一个程序，责任是解释执行字节码文件，有跨平台的性质。

**作用：**

- 解释执行字节码文件
- 管理内存中的对象分配，垃圾的自动回收
- 优化热点代码，提高执行效率



**虚拟机的结构：**

- 类加载子系统：类加载器
- 运行时数据区：堆、程序计数器、本地方法栈、方法区（jdk7开始渐渐移除永久代，开始使用元空间，元空间位于本地内存中，而不是jvm内存中）
- 执行引擎：解释器、即时编译器、垃圾回收器
- 本地方法：本地接口

## jvm在jdk6-8之间内存区域有什么不同？

1. **方法区位置不同**

   jdk6时，方法区位于堆中，但是jdk8开始方法区变为元空间位于操作系统的直接内存上。

   

2. **字符串常量池位置不同**

   jdk7之前，运行时常量池逻辑包含字符串常量池，hotspot虚拟机对方法区的实现为永久代。

   jdk7时，字符串常量池被从方法区拿到了堆中，运行时常量池剩下的东西还在方法区（永久代）。

   jdk8以后hotspot虚拟机移除了永久代用元空间取而代之，字符串常量池位置不变，还是在堆中。

### 疑惑

**使用元空间代替永久代的原因**

- **提高内存上限**
- **优化垃圾回收策略**：永久代在堆上，垃圾回收机制一般使用老年代的垃圾回收方式，不够灵活，使用元空间之后单独设计了一套适合方法区的垃圾回收机制。



**为什么字符串常量池从方法区移动到堆中？**

- **垃圾回收优化**：字符串常量池的回收逻辑和对象的回收逻辑类似，内存不足的情况下，如果字符串常量池中的常量不再被使用就可以被回收，方法区中的类的元信息回收逻辑更复杂一点，移动到堆之后，就可以利用对象的回收器对字符串常量池进行回收。
- **让方法区大小更可控**：一般在项目中，类的元信息不会占用特别大的空间，所以会给方法区设置一个比较小的上限，如果字符串常量池在方法区中，会让方法区的大小不可控，如果线程中创建了很多字符串常量，那么方法区就会变得很大。
- **优化intern方法**：想一下，字符串常量在创建时是不是本来就是一个对象，在这个基础上来思考，jdk6版本中使用intern方法，会把第一次遇到的（也就是你想要它存在的字符串常量）字符串实例复制到永久代中的字符串常量池中。需要复制的字符串实例本来就是在堆空间中的，这里也就多出了一个复制的过程，怎么优化？jdk7及之后的版本中直接把字符串常量池移到堆中就可以了，省去了一个复制过程。字符串保存在堆中，把字符串的引用（意思就是在堆存创建的对象的地址）放入字符串常量池中，减少复制操作。

## 什么是类加载器？

### 作用

类加载器负责在类的加载过程中将字节码信息以流的方式获取并加载到内存中。简单说就是将java类的字节码（.class文件）加载到jvm中（在内存中生成一个代表该类的Class对象）

### 分类

- **启动类加载器**：加载核心类
- **扩展类加载器**：jdk9之后改名为平台类加载器，加载扩展类
- **应用程序类加载器**：加载应用classpath中的类
- **自定义加载类**：重写findClass方法就行



## 什么是双亲委派机制？

**前提知识：**类加载器有层级关系，上一级称为下一级的父类加载器。

由低到高：

自定义类加载器 -----> 应用程序类加载器 -----> 扩展类加载器 -----> 启动类加载器

这里又想到为什么自定义类加载器的父类加载器是应用程序类加载器呢？

这里的原因就在于类加载器的无参构造是使用的抽象父类的无参构造方法

```java
public abstract class ClassLoader {
    // 你自定义的类加载器如果写成
    // public MyCL() {}          // 编译器会默认插入 super()
    // 或显式  super();
    // 都会走到这里：
    protected ClassLoader() {
        this(checkCreateClassLoader(), getSystemClassLoader());
    }

    // 真正给 parent 赋值的构造
    private ClassLoader(Void unused, ClassLoader parent) {
        this.parent = parent;          // 这里完成赋值
        ...
    }
}
```

**自定义类加载器的 parent 是在 `ClassLoader()` 无参构造里通过 `getSystemClassLoader()` 一次性绑定的，绑的就是 JVM 启动时建好的应用程序类加载器。**



**双亲委派机制**指的是：当一个类加载器接收到加载类的任务时，会向上查找是否加载过，在由上向下进行加载，由上向下加载的过程中，类加载器会检查自身是否可以加载（需要加载的类是否在自己的加载目录中），如果不符合，则交给下面的类加载器进行加载。



### 双亲委派机制有什么用呢？

- 保证类加载的安全性
- 避免类的重复加载





## 如何打破双亲委派机制？

首先需要了解双亲委派是在哪实现的

`ClassLoader`抽象父类中包含了这四个方法

``` java
//String name 表示的是需要加载的全限定类名

//类的加载入口，提供了双亲委派机制。内部会调用findClass方法 重要！！
//这个方法里面会判断这个类是否被当前的类加载器加载过
public Class<?> loadClass(String name)

//由类加载器子类实现，获取二进制数据调用defineClass，比如URLClassLoader会根据文件路径去获取类文件中的二进制数据 重要！！
//简单说就是检查是否可以加载name这个类，可以就给出这个类，不可以就留给下一层类加载器加载
protected Class<?> findClass(String name)
   
//做一些类名的校验，然后调用虚拟机底层的方法讲字节码信息加载到虚拟机内存中
protected final Class<?> defineClass(String name, byte[] b, int off, int len)
    
//执行类生命周期中的连接阶段，默认是不会执行的，需要传参开启
protected final void resolveClass(Class<?> c)
```

## jvm中有哪些引用类型？

- ### 强引用

  java中默认引用关系就是强引用，即是对象被局部变量、静态变量等GC root[^1]关联的对象引用，只要这层关系存在，普通对象就不会被回收。

  [^1]: GC root也叫做根对象，哪些算作根对象呢？线程`Thead`对象，引用线程中栈帧中的方法参数、局部变量等；系统类加载器加载的`java.lang.Class`对象；监视器对象，用来保存同步锁synchronized关键字持有的对象；本地方法调用时使用的全局变量。

- ### 弱引用

  弱引用的整体机制和软引用基本一致，区别在于弱引用包含的对象在垃圾回收时，不管内存够不够都会被直接回收掉，弱引用主要在`ThreadLocal`中使用到。
  
- ### 软引用

  软引用相对于强引用是一种比较弱的引用关系，如果一个对象只有软引用关联到它，当程序内存不足时，就会将软引用中的数据进行回收，软引用主要在缓存框架中使用到。

  

- ### 虚引用

  虚引用又叫做幽灵引用、幻影引用，不能通过虚引用对象获取到包含的对象。虚引用唯一的用途是当对象被垃圾回收器回收时可以接受到对应的通知。直接内存中为了及时知道直接对象不再使用，从而回收内存，使用了虚引用来实现。
